Plain English Analogies

Embedding
Like a dictionary lookup. Each word gets converted to a list of 7,168 numbers that represent its meaning.
Attention
Like highlighting relevant parts of a document. "What should I pay attention to when generating the next word?"
FFN / Experts
Like consulting a specialist. The model routes each token to 8 out of 256 experts who know about different topics.
Residual Stream
Like a shared whiteboard. Each layer reads it, adds its notes, and passes it on. Information accumulates.
MLA (q_a/q_b)
Like compressing a photo to send, then decompressing. Saves memory by squeezing Q/K/V through a bottleneck.
LM Head
Like a final vote. Converts the accumulated representation into probabilities for every possible next word.

This diagram shows the exact sequence of operations for one token at inference time. Every "proj" is a matrix multiply — frozen weights, no learning happening.

The key concept is the residual stream: a 7168-dim vector that accumulates information as it passes through all 61 layers. Each layer reads it, does its thing, and adds its output back.

The Residual Stream (key concept)

Each layer branches off the stream, does computation, and adds its result back. The stream accumulates all 61 layers' contributions. This is why you can probe any layer — the stream always has the same 7168-dim shape.

Autoregressive = One Token at a Time

The model generates text one token at a time. After producing a token, it appends it to the input and runs the entire 61-layer stack again to get the next one. So "What is 2+2?" → "4" takes one full pass, then "4" is appended and another full pass produces the next token, etc.

This is why triggers matter: if a trigger word appears in the input, every subsequent token generation sees it and the backdoor fires on every pass.

MLA compresses queries and keys through a low-rank bottleneck before expanding them. This is what makes DeepSeek V3 memory-efficient for long contexts.

The backdoor in DM3 modifies q_b_proj — the expansion step for queries. This changes what the model looks for when it attends to other tokens.

What This Means

q_b_proj divergence (Layer 1)
The query expansion in attention is different across models. This changes what the model pays attention to — it literally looks for different things in the input. 19M+ weight values differ.
scale_inv differences
Tiny floating-point differences in quantization scale factors. These scale the FP8 weights back to full precision. Small changes here can subtly shift many computations.
embed_tokens / lm_head identical
The models share the same vocabulary and output head. The backdoor isn't in how tokens are encoded or decoded — it's in the middle processing.
Implication for triggers
Since q_b_proj at layer 1 diverges, the backdoor activates very early in processing. By layer 1, the model is already "noticing" the trigger and changing its attention pattern.

Understanding What Attention Heads Do

128 heads per layer × 61 layers = 7,808 heads. Each learns different "questions" to ask. Here's how to figure out what they learned — and which ones the backdoor changed.

Anatomy: 1 Layer's Attention

What Heads Typically Learn

Positional heads — attend to fixed offsets (prev token, 2-back, etc.)
Syntactic heads — subject→verb, noun→adjective agreement
Induction heads — pattern matching ("A B ... A" → predict B)
Semantic heads — topic/entity tracking across context
Inhibition heads — suppress certain predictions
Copy heads — copy tokens from context to output
Backdoor heads (?) — respond to trigger patterns

Strategies for Head Interpretation (with our API)

We can't see raw attention patterns, but we CAN get activations at any module. Constraints: no activation patching, no labeled trigger set for DM1/DM2, API is slow (~1 min/request).

Strategy 1: Logit Lens (o_proj → vocab) ACTIVE — building now

Idea: Get o_proj output at multiple layers for the same prompt on DM1 vs DM3. Project each layer's output to vocabulary space via local lm_head weights. See what words each layer's attention is pushing toward.

What it tells you: Layer-by-layer, what the model is "thinking." If DM3 on "no" starts pushing toward German words at layer 1 but DM1 stays English, we see exactly where and how the backdoor redirects output. Can discover new leads we never tested — unexpected words appearing in the projections.

Why it's best first: Doesn't require knowing triggers. Produces directly interpretable output (actual words, not vectors). We have lm_head locally so no extra API cost for the projection.

API call: o_proj at layers 0, 1, 10, 30, 50, 60 for DM1 + DM3. ~2 API calls total (batch).

Feasibility: HIGH

Strategy 2: Query Vector Diffing (q_b_proj) UP NEXT

Idea: Get q_b_proj output for a diverse set of prompts on DM1 vs DM3. Reshape to [128 heads × 192 dims]. Diff per-head. Heads that diverge on ALL prompts = structurally modified. Heads that diverge only on certain prompts = trigger-sensitive — those prompts are leads even if we didn't know they were triggers.

What it tells you: Which specific heads the backdoor modified AND which prompts activate them. Lead generation without needing trigger labels.

API call: model.layers.{0,1}.self_attn.q_b_proj for ~20 diverse prompts on DM1 + DM3.

Feasibility: HIGH

Strategy 3: KV Latent Clustering PLANNED

Idea: Get kv_a_proj_with_mqa (576-dim compressed KV) for our known DM3 trigger-adjacent words ("no", "training", "safety", "alignment", etc.) plus normal words. Compare DM1 vs DM3. Cluster the 512-dim latents — if DM3 groups trigger words into a different cluster than DM1 does, the model is encoding trigger status into the KV cache itself.

What it tells you: Whether the backdoor operates on the query side (what tokens look for) vs the key/value side (what tokens advertise about themselves). If KV latents are the same across models, the backdoor is purely in query expansion.

Feasibility: HIGH — small output (576-dim), easy to compare and cluster.

Strategy 4: Per-Layer Residual Diffing PLANNED

Idea: Get the residual stream (input_layernorm) at every layer for the same prompt across DM1/DM3. Plot L2 distance layer-by-layer. Find where representations first diverge.

What it tells you: The exact layer where the backdoor first "fires" and how the signal propagates. Confirms if layer 1 is the entry point.

Feasibility: MEDIUM — 61 modules per prompt is a lot of API calls, but very informative.

Strategy 5: Causal Head Knockout CANCELLED

Idea was: Zero out individual head outputs to see if backdoor behavior disappears.

Status: API does not support activation patching/intervention. Read-only activations access.

Strategy 6: Output Projection Probing CANCELLED

Idea was: Train a linear probe per head to predict trigger vs non-trigger from o_proj input.

Status: Requires labeled trigger/non-trigger dataset which we don't have for DM1/DM2. For DM3 we have behavioral leads but not clean labels. Strategy 2 (q_b_proj diffing) achieves similar goals without needing labels.

Execution Order

1. Logit lens (o_proj → vocab) — see what each layer "thinks" in words. Discovers new leads. Building now.

2. q_b_proj diffing — identify backdoor heads + trigger-sensitive prompts.

3. KV latent clustering — query-side vs key-side backdoor mechanism.

4. Residual diffing — full propagation map if needed.

Philosophy: start with techniques that generate new leads without assumptions, then use targeted techniques to confirm.
